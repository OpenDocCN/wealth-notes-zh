## 0、出海工具站高手是怎样快速上线一个网站的？

出海工具站高手究竟是怎样快速上线一个网站的？其实我也不知道！因为我是工具站新手。在哥飞社群中，时不时都会听说有些高手在机会突然到来时，一两个小时内上线一个网站，通过快速上线网站布局关键词，抢占流量制高点。

虽然我不知道高手是怎样快速上线一个网站，但是，网站上多几个后，我开始总结一个网站上线的流程，按照流程，我一样可以做到一两个小时上线一个网站，不管是 NextJS 网站，亦或 WordPress 网站。这也是哥飞经常和我们说，要多上站、多练习的原因，实践多了，你自自然然会熟悉一套适合你的上线流程，你也可以把它总结成一个上线 SOP。而且，这个 SOP 不仅适合你上线工具站，还适合你上线游戏站，或者其它类型的网站。

接下来，给大家讲讲我平时上线 NextJS 网站的流程。这个流程不一定和高手的上线流程一致，但却有可能更加适合我们小白或新手。

## 快速上站
- 1.1 在你熟悉的域名商注册域名
当你找到一个需求或者某个新词突然出现的时候，你要做的第一件事不是开发网站，而是在确定是否做这个需求或新词之后，马上注册相关域名。选择你已经准备好的熟悉的一两个域名商注册即可，Namecheap 也行，阿里云也行，其它也行，尽量不要因为其它域名商费用更低而换注册商，这样有助于你快速注册你想要的域名，后续设置域名 DNS 和域名解析时也会更加熟悉。

### 1.2 更换 Cloudflare 的 DNS
注册好域名之后，你要做的事情仍然不是开发网站，而是先在 Cloudflare 中添加网站，并把域名的 DNS 服务器更换成 Cloudflare 的 Nameservers，这样后续域名的解析、缓存、SSL、Email 等都可以在 Cloudflare 中进行。为什么要把 DNS 切换放在前面，这是生财航海教练良辰美的一个经验，因为域名修改了 DNS 后，不一定马上生效，有时几分钟内，但有时时间会更长。如果没有提前设置好，正式部署的时候可能就会因为等待 DNS 生效而耽误时间。

### 1.3 通过 AI 生成首页文案
对于新上线的网站，想通过谷歌 SEO 获取流量，首当其冲的是网站内容和关键词布局，因此，我会围绕某个需求或新词对应的关键词，让 ChatGPT 出首页文案，如果你熟悉 Claude 或 Gemini，也可以使用其它 AI 模型生成首页文案。最好把文案保存成 Markdown 文件。

### 1.4 在编程 IDE 完善网站首页
通过 Cursor、Kiro 或其它 IDE 根据首页文案的 Markdown 文件完善首页的内容（当然，你也可以使用 Claude Code），如果你对页面 UI 有所要求，也可以把“2.4 完善网站 UI 风格”提前放到这一步中。完善好首页内容后，注意让 AI 检查并完善页面的 SEO 相关要素。

### 1.5 生成网站 sitemap.xml、robots.txt 和 llms.txt 文件
在本地编译启动网站测试无误后，直接让 AI 根据网站当前提供的页面和功能生成 sitemap.xml、robots.txt 和 llms.txt 文件。如果你的网站需要在这个阶段连接数据库才能正常访问的话，可以把“2.2 对接 Supabase”这一步提前放到这一步中。

### 1.6 提交代码到 GitHub
通过 Cursor 或 Kiro 的 Git 功能把代码提交到 GitHub 中，当然你也可以通过命令行进行提交，也可以通过 GitHub Desktop 辅助提交。

### 1.7 在 Vercel 中导入网站做部署
到了这几步，网站已经可以部署上线了！可能你会说，网站功能还没开发好呢！事实上，到了这一步，网站不仅功能还没做好，登录也还没做好，甚至数据也还没对接，UI 也还没完善。但是没有关系，你这个阶段的目标可能是尽快上线，争取排名和流量。因此，你可以在 Vercel 中导入直接导入你 GitHub 中对应的项目，直接部署上线。

### 1.8 做域名解析
在 Vercel 中部署成功后，Vercel 会给你提供一个二级域名（这个不是你的域名）。你需要添加你的域名，一般添加 xxx.com 和 www.xxx.com 这两个，分别在 Cloudflare 中做 A 解析和 CNAME 解析。

这个时候 Cloudflare 中添加的网站的 DNS 往往已经生效，我除了做域名解析之外，一般还会同步添加缓存设置、邮件系统设置等。

域名解析好之后注意把其中一个域名（比如：www.xxx.com）以及 Vercel 提供的二级域名都通过 301 重定向到另一个域名中（比如：xxx.com）。

这个时候，你的网站已经上线了！检查网站打开是否无误，检查 Sitemap、robots.txt 和 llms.txt 是否能正常打开。

### 1.9 在 GSC 和 Bing Webmaster 添加网站
网站上线后，先在 GSC 中添加网站，并提交 Sitemap 和首页；然后在 Bing Webmaster 通过导入 GSC 添加网站，同样提交 Sitemap 和首页。

### 1.10 添加 GA 和 Clarity 跟踪代码
最后，把网站添加到 Google Analytics 和微软的 Clarity 中，并把跟踪代码通过 Claude Code 或其它 AI 编程工具添加到你的网站中，提交 GitHub，并在 Vercel 中重新部署。

## 工具站出海的第一个坑：域名注册需要注意什么？

我们先讲讲每个出海网站都绕不过的话题：域名注册。

可能你会说，域名注册这种简单到不能再简单的事情有什么好说？我想，这有可能是因为你是出海高手或老手，但是你可能不一定能想到许多出海新手可能连在哪里注册域名都不知道。

接下来着重讲讲出海过程中域名注册有哪些坑，大家下次注册域名碰到时候别往里跳。

### 1.1 你注册的是出海网站域名，不是国内网站域名
如果你做的是出海网站，不管是工具站、游戏站亦或内容站，你都不需要备案。但是，你不能注册“.cn”或“.com.cn”这些国内域名，“xxx.中国”或“中文.com”这类中文域名更加不要注册。而且，你使用的服务器必须是海外的，美国、欧洲、港澳台都可以，否则，你的网站无法正常解析和访问。当你使用 Vercel 或 Shopify 的时候，由于这些平台的服务器是在国外，所以你的域名一般都可以正常解析到这些平台上。

如果你做的是国内网站，请老老实实按照我们国内的政策，使用国内的服务器并做备案，域名使用“.com”、“.net”、“.cn”、“.com.cn”等等都可以，但是，你别想着拿“.us”的域名去做国内备案。

### 1.2 避开二级域名代理商
注册域名的时候，尽可能选择顶级（一级）注册商，阿里云、腾讯云、GoDaddy、Namecheap 等都是顶级域名商，而在其下还有许多二级代理商。二级代理商往往是一些个人或小型工作室，存在诸多风险，ChatGPT 就这个问题做了以下回复：

| 风险点 | 说明 |
| :--- | :--- |
| 账户与所有权隔离 | 在 Reseller 体系里，你的域名账户实际托管在上游 Registrar 名下；若代理商倒闭或纠纷，迁移成本高。 |
| 价格与隐形费用 | 代理商通常批量拿低价后自行定价，续费、转移、隐私保护等附加费用可能比正规 Registrar 更高或不透明。 |
| 服务与合规 | Reseller 不直接受 ICANN《2013 RAA》监管，停靠期限提醒、数据准确性校验、争议处理等合规要求执行力度参差不齐。 |
| 技术与安全 | 许多 Reseller 不提供 Registrar 级别的安全功能（Registrar Lock、DNSSEC、批量 API、二次验证），DNS 列表或联系人修改可能需要人工工单，效率低且易出错。 |
| 极端情形下的资金和数据风险 | 历史上出现过 Reseller 跑路或被上游 Registrar 终止 API 资格的案例，最终用户需通过繁琐的“批量迁移”才能取回域名。 |

因此，即使二级代理商给你多优惠的注册价格，最好别贪图小便宜。我个人使用最多的是 Namecheap、Spaceship 和阿里云，Namecheap 及其旗下的 Spaceship 可以注册绝大多数域名包括 ai 域名，Spaceship 往往更加实惠，阿里云作为补充，这些平台都有各自优势，并且都可以做域名交易。

前面我们提到这点，优先在你熟悉的域名商注册。原因有四：

- 一是，上面提到的要回避二级域名代理商；
- 二是，在一个新的域名平台注册的时候，往往需要填写大量的个人资料和注册信息，需要花费不少时间；
- 三是，修改 DNS、解析域名以及后续转移域名时有可能会因为不熟悉这个域名平台而费一番周折；
- 四是，域名注册太分散，不利于后续的管理和续费。

### 1.4 域名续费和域名转出可能是个坑

在上面 3 个域名商的价格截图中，你可能发现有些域名的价格特别便宜，比如，“.online”域名在阿里云只需¥8/年，在 Namecheap 只需$0.98/yr。

但是，可能你没注意到，它旁边有一个域名的续费价格，Namecheap 显示的续费价格是$27.98/yr（图片没有显示），而阿里云的续费价格是¥99/年，这样意味着你明年需要花更多的钱进行续费。

可能你会想，能否明年让它自动过期了，再重新按优惠价购买？那你就想多了，域名过期后有 30 天续费宽限期，而后有 30 天赎回期，再之后才是准备删除。在这段时间内，你对应域名的网站是无法打开的，而且即使你等到那个时候重新注册，万一你的域名很抢手，就有可能会被其他人抢注。

另外，某些平台把域名转离当“割韭菜”，会对买家收转出费用。

### 1.5 不需要额外购买域名邮箱

在你购买域名的时候，有些域名商会赠送你域名邮箱，有些会建议你额外购买域名邮箱。但是，这些都没有必要，直接使用Cloudflare的邮件功能就可以了，Cloudflare Email Routing 可以帮你通过Gmail等邮箱转发接收邮件，但不含发信；若要发信需配合 Gmail SMTP / Amazon SES / SendGrid 等。

### 1.6 不需要额外购买 SSL

现在的网站没有配备 SSL，基本上是无法见人的，你平时看到的网站基本都是“https”开头而不是“http”开头，就是因为这些网站都使用了 SSL。

阿里云有卖 SSL 和 CDN 这些服务，价格还不低。但是，有了 Cloudflare，你完全没必要购买 SSL，直接使用 Cloudflare 提供的 SSL 即可（后面讲 Cloudflare 相关的部分时再细说）。如果你把网站部署到 Vercel 上，Vercel 也会帮你自动生成 SSL。

### 1.7 域名选择有讲究

域名优先选择“.com”的域名，这是大众的普遍认识。如果“.com”的域名已经被注册，可以考虑“.net”、“.org”，如果这两个也被注册了，还可以考虑“.ai”，但是“.ai”的域名很贵，要求一次性支付 2 年，在 Namecheap 中 2 年大概要$160 左右，Spaceship 便宜一些，2 年大概要 138 美金。

如果你做的是出海网站、选择其它后缀的域名也是可以的，比如刘小排老师的 Raphael AI 使用的域名是 raphael.app,再比如艾逗笔的 MCP 导航网站的域名是 mcp.so，最近爆火的 Wplace 的域名是 wplace.live，再比如 Bolt.new、v0.app、manus.im 等等。

从谷歌 SEO 的角度，使用什么域名后缀并不是影响排名的原因，并且，许多域名后缀，包括“.ai”、“.app”、“.io”等，在海外用户群中的认知度还是挺高的。

### 1.8 域名是不是越短越好？

从有利于用户记忆和识别的角度，域名当然是不要太长更好，是大众熟知的单词则更好。但是，互联网发展到现在，许多常见的单词已经很难注册到。所幸，哥飞老师给我们提供了一个新的思路：做新词网站，把过去没什么人搜索但近期突然搜索量暴增的新词注册成域名，举全站之力推这个关键词，那么，我们的网站就有机会拿到排名拿到流量。而且，从SEO 的角度，谷歌并不会因为你的域名太长而不给你流量，只要你的网站内容能满足用户的搜索意图，就有可能获得流量。

自从 AI 时代到来之后，这样的新词层出不穷，比如最近爆火的 Nano-Banana，有不少朋友注册了 nanobanana 或 nano-banana 的域名并快速上线相关的生图网站，获取到不少流量。对于这种词，建议优先注册没有“-”的域名，符合用户输入习惯，谷歌也会自行识别这是两个单词，域名后缀前面已讲，优先“.com”、“.ai”、“.net”、“.org”这些，其次才是其它。

### 1.9 小心你的域名有侵权隐患

注册域名的时候，对于一些品牌名称或知名人物名称，建议谨慎注册。我曾经就这个问题和 ChatGPT 讨论过：

**TL;DR**
- 美国联邦商标法(Lanham Act)的主战场仍在“美国境内的商业使用”。
- 如果你把网站、付款、物流和营销都明确排除美国市场,在最新的*Abiiron v. Hetronic*(2023)判例之后,USPTO 注册商标所有人要想在美国法院起诉你,难度显著增大。(wilmerhale.com)
- 但风险≠0:
  - UDRP域名仲裁全球适用——无论你是否向美国销售,只要被视为“恶意注册并使用”,域名仍可能被强制转移。
  - “境外也能影响美国”——若网站对美国用户保持可见、可下单或生品牌混淆,仍可能构成“境内使用”,让权利人有机可乘。
  - 其他国家/地区的商标法仍然适用;中国、欧盟等市场若有在先商标,你同样面临侵权或异议。

全球未注册但有使用的商标 用 WIPO Global Brand Database + Google/社媒搜索其名字+商品/服务 在UDRP中,“未注册但已被公众识别”的个人名字也可构成可保护标识。

如果结果:
- “无注册记录→法律风险主要来自“姓名权/人格权”与“未注册商标使用”;”
- “有注册→视同商标在先,风险加倍,需要谨慎出售策略。”

- **UDRP 视角:社媒网红名字≠自动安全**
  - UDRP 小组已多次支持网红/艺人拿回姓名域名。
  - 例如知名 TikTok 红人 Charli D'Amelio 成功通过 UDRP 取回<charlidamelio.shop>等域名。(wipo.int)
  - 核心理由:其姓名已在全球范围作为商品与服务标识,具有“共同法(common-law)商标”权。
  - 判断基准:
    - 域名与姓名相同或近似;
    - 你是否拥有“合法权益”(如本来就叫此名并且用于真实个人站点);
    - 注册/使用是否带恶意(如高价转售或误导流量变现)。

如果你注册的域名名称是之前就已经被注册了的商标,或者“未注册但已被公众识别”的个人名字,都有可能存在侵权风险而被收回,甚至会有被仲裁的风险。但是,有时机会与风险并存,大家各自掂量。

## 通过 AI 编程打造你的工具站 2.0 落地页

前面我们已经使用 ChatGPT 的 “Deep Research” 生成工具站首页的文案，接下来，我们正式进入开发一个网站的环节。

这个时候，你可能会想，现在 AI 编程那么牛了，我直接让 Claude、Cursor 或者 v0 帮我从无到有生成一个不就得了。

你真这么想，那就太天真了！如果你是一名编程高手，那么你有 N 种办法上线一个网站，包括从 0 到 1；但是如果你和我一样只是一名新手，那你最好别想着从 0 到 1 开发一个网站。你首先要做的是找到一个契合你的需求的框架或模板，而且最好是开源的。

这样的好处是，你可以站在高手的肩膀上，节省大量开发时间，从而把你的开发精力集中到你的需求实现上。比如 NextJS 框架你可以直接选择使用刘小排老师在深海圈提供的 raphael-starterkit-v1，或者艾逗笔的 ShipAny，或者让 ChatGPT 给你推荐 Github 上其它开源框架，再比如 PHP 你可以考虑 WordPress 框架。这些框架一般都已经集成好用户系统、注册登录、支付系统、积分系统等等。而你想把这些功能从 0 到 1 开发出来，实际上并不那么容易，即使现在 AI 编程很强，更大的可能是漏洞百出、暗藏隐患。

## 网站上线前的细节工作和注意事项

我们通过 ChatGPT 的 “Deep Research” 生成网站首页的文案，并通过 Claude Code 把这些文案放进一个开源的 NextJS 网站中。到了这一步，如果你的网站的主关键词是近期某个热点，那么，就要尽快上线，即使网站核心功能还没开发、注册登录还没完善、支付接口还没对接，都没有关系，先把网站上线争取搜索排名和 SEO 流量。但是，正式上线之前，你仍然需要做好一些细节方面的准备工作，否则后面可能会影响你的网站流量，甚至造成网站前功尽弃。

那么，有哪些细节需要注意呢？我总结以下这些点，如有遗漏，烦请各位出海同胞留言补充。

### 1、检查并确定网站域名

确定网站域名？这句话听起来像是废话！但是，前段时间刚好有一个品牌升级的案例，Turbolearn AI 进行品牌升级，把域名从“turbolearn.ai”切换为“turbo.ai”，结果近期流量大跌：

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_56_0.png)

哥飞在社群中给我们分析了这个案例，并详细告诉我们怎样分多个步骤进行品牌升级。Turbolearn AI 作为一个百万级的大站，它背后的团队可能有很多办法可以让网站的流量回到从前的高度，但是对于我们独立开发者，如果网站因为更换域名造成流量暴跌，有可能是致命而不可挽回的损失。因此，上线一个网站的时候，一定要再三确定使用哪个域名，并且后续不要轻易更换。

确定网站域名之后，你还要确定究竟使用哪个域名？这句话听起来又像是废话！但是，你必须要清楚地明白：“xxx.com”这个域名和“xxx.net”这个域名是两个完全不同的域名。而且，你还要清楚地明白：“xxx.com”这个域名和“www.xxx.com”这个域名也是两个不同的域名。你选择的是哪个，将决定你后面进行域名解析和跳转的策略。

另外，你还要检查一下域名是否开启域名隐私保护，防止被其他人随意查看你的个人信息，以及查看域名是否有曾经被搜索引擎惩罚的记录。

### 2、确认 SSL

与域名相关的 SSL（你在地址栏里看到的 https://），如今几乎成了现代网站的标配。这是主流浏览器与搜索引擎对安全站点的硬性要求，也向访客表明“网站已开启加密传输并通过身份验证”，从而防止中间人窃听、数据篡改或钓鱼仿冒。因此，在正式上线前你要先决定用哪家的 SSL 证书。
- Cloudflare：域名接入后，一键即可启用 Universal SSL。
- Vercel：添加并解析自定义域名后，平台会自动为你签发证书。
- Let's Encrypt：最流行的免费 CA，配合 ACME 客户端（如 Certbot、Caddy、Traefik）即可自动续期。

你需要注意的是，千万别想着先上线一个 http 的网站，后面再切换成 https，因为 “http://xxx.com” 和 “https://xxx.com” 在搜索引擎眼里是两个不同 URL，前者走 80 端口，而后者却走 443 端口。切换后需要做 301 重定向、更新站内链接、重新索引等一系列迁移工作，代价远高于一开始就上 HTTPS。

### 3、确定网站框架

网站上线之前，甚至开发之前，最好先想清楚你的网站是什么类型的网站，打算使用什么框架？如果是电商网站，我大概率会选择 WordPress，因为这个生态所配备的模板、插件，以及 WooCommerce，足以满足各种类型的电商网站和 CMS 网站，你甚至不需要做任何开发就可以上线一个功能齐备、UI 上佳的网站；如果你嫌部署一个 WordPress 网站麻烦，可以选择那些已经安装好 WordPress 网站的服务器或空间，也可以直接选择使用 Shopify，像开网店一样经营一个网站；如果是订阅网站，建议按小排老师和哥飞老师推荐的 NextJS 框架，并采用一个契合你的需求的开源模板，结合 Supabase 数据库，部署到 Vercel 上，这样你的网站上线难度会大大降低。

确定好网站框架，开发上线后，建议别轻易更换网站框架和模板，不要把网站从 WordPress 框架改成 NextJS 框架，也不要把网站风格从一个模板换成另一个模板，这些都有可能会造成网站前端页面结构大幅改动，这在 SEO 上是大忌。

前不久我上线了一个网站，一开始没想好做什么，就随便用已经部署好的多站点 WordPress 发布了一个首页，没多久就出现 24 小时有近 1 万曝光，但也就是那个时候，我把网站切换成 NextJS 框架，结果第二天开始流量大跌：

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_60_0.png)

### 4、使用 AITDK 检查并确认 SEO 相关要素

你通过 AI 生成网站首页的文案，并通过 Claude Code 把这些文案放进 NextJS 网站后，建议使用哥飞的群友开发的 AITDK 这个 Chrome 插件，检查并确认网站的 “Title”、“Description”、“H1”、“H2” 等关键 SEO 要素的内容是否正确，尽可能让这些内容与你后续网站提供的功能和服务相符。尤其重要的是，一定要确保整个页面只有一个 “H1” 标签，并且在这个 H1 中放置网站首页的核心关键词。我之前看到过有网站把 Logo 的文字设置成 H1，然后在首页 Hero 区域又增加了一个 H1，这样不利于搜索引擎识别你的网站页面的核心关键词。

### 5、删除冗余页面和内容

你开发上线一个网站的时候，如果选择了某一个开源的框架，或者在你过去已有的项目基础上做二次开发，那你在上线之前需要确认一下，除了网站首页之外，网站中是否还存在其他页面。这些页面的内容是否和你准备上线的网站不相关，需要修改或删除。否则，网站上线后，谷歌有可能会索引到这些与你网站不相关的页面，甚至索引到这些和原有网站重复的页面。

### 6、检查多语言页面

你在开发上线一个网站之前，还需要确定你的网站第一版是否上线多语言。如果考虑多语言，那你要认真做好每一个多语言页面，如果只是随便翻译应付一下，那就不如先不要上多语言。如果不考虑多语言，那你要注意你选用的模版中是否包含多语言，包含的话要隐藏或删除，或者干脆先选择单语言的模板，后续需要再另行增加多语言。

### 7、检查 API Key 是否硬编码

这一点很重要！有时候我们通过 AI 编程开发调整 API 的时候，可能会把 API 提供给 AI，而 AI 生成的代码可能把你的 API Key 写在代码中，这样有可能会造成你的 API 密钥泄露，带来或大或小的损失。为保险起见，上线之前，你需要再三确认 API Key 是否只保存在配置文件中，并且，你还需要通过 Cursor 或其它你使用的 IDE 全局搜索你的 API Key，查看并确保你的所有代码文件都没有 API Key 的硬编码。

## AI 把我的代码越改越乱! 怎么办?

在 AI 编程的圈子里，时不时都会看到有人无比郁闷地说：“崩溃了！AI 把我的代码越改越乱！”

每当看到这样的话，我总想说，这不是很正常的吗？你让一个有一定经验的程序员帮你做项目，一样有可能把项目搞砸啊。

对于我们编程小白或新手来说，AI 已经成为我们开发项目实现想法的好帮手，但是，我们怎样才能尽可能让 AI 不要把代码改乱？怎样应对 AI 把代码越改越乱呢？

今天，我重点说两个点：

- 你自己要想办法把话说清楚
- 你要给自己留有余地

### 第一点：你自己要想办法把话说清楚

网上流传着一个段子：

老婆让老公去买一堆东西，比如“买一斤排骨、半斤韭菜、再买点蘑菇，如果看到西瓜，就买一个”。

结果，老公只带回来一个西瓜。

老婆很生气，问老公：“我让你买那么多东西，你怎么就买了个西瓜？”
老公回答：“你不是说如果看到西瓜，就买一个吗？我一进超市就看到西瓜了。”

我们在让 AI 帮我们编程也一样，你要尽可能把你的需求讲清楚，尽量不要让 AI 去猜测你的想法。对于较为复杂的需求，我往往会让 AI 先出一份方案，和它再三确认方案并保存成 Markdown 文件后，再让它根据文件中的方案进行开发。

参加刘小排老师的深海圈之后，我有机会见识到业界顶尖人士怎样通过 AI 编程，可以说小排老师把“想办法把话说清楚”做到了极致。比如他演示开发一个“吵架包赢”的网站，通过 AI 生成内容并实现语音朗读，他会先查看对应 API 的文档，然后通过 Postman 调试每个 API，调试通过后，才把对应 API 的文档、API 示例、示例输入和示例输出都一并发给 AI，这样 AI 往往能更加明确他的需求，在他圈定的范围内给出更加合理的代码。用他的话说：“你不要让 AI 抽卡！”

### 第二点：你要给自己留有余地

第二个我想说的就是备份。这件事情做起来很简单，但也很容易被很多人忽略，这也是为什么我们经常看到有人说“崩溃了！AI 把我的代码越改越乱！”的重要原因。如果你已经把 AI 编写的代码的每个重要成果和关键步骤都做好了保存和备份，为什么还要担心 AI 把你的代码改乱呢？你完全可以放开手脚让 AI 尽情开干，大不了回撤到上一个正确的位置，然后让 AI 调整姿势，重新开始。

## 把代码推上 GitHub 的最佳姿势，为 Vercel 一键部署铺路

作为全球最大的代码托管平台，Github 已经汇集了超过 1 亿开发者，它不仅提供云端代码备份，更汇聚了全球程序员的智慧和结晶，你可以在上面找到各种编程语言的开源代码。有传闻称 OpenAI 早期的“编程模型”就是通过其背后金主微软提供的 GitHub 代码进行训练的，可见这个平台在编程领域有举足轻重的地位。

通过 GitHub 提交和托管代码，对于程序员来说是家常便饭，网上也有大量相关的文章。但是，作为技术小白或编程新手，我们未必习惯看似简单的 Git 提交命令。今天，我们尝试通过另一种对新手最佳的姿势，把我们前面准备好的代码推送到 Github 上。

在开始之前，请你先安装并配置 Git，然后安装 GitHub Desktop 这个图形界面。如果你不懂怎样安装，请查看网上的相关教程，或者直接咨询 ChatGPT 或 Deepseek。最后，你还需要有一个编程的 IDE，Cursor 或 Visual Studio Code 或其它 IDE 都行，本文我们将以 Cursor 为例。

前面我们已经通过 ChatGPT 的“Deep Research”生成网站首页的文案，并通过 Claude Code 把这些文案放进一个开源的 NextJS 网站中，现在，我们讲一下怎样把代码推送到 Github 中，为后面的 Vercel 一键部署铺路。

### 1、检查.gitignore 文件

在提交之前，我们先检查一下.gitignore 文件，这是一个纯文本配置文件，用来告诉 Git “哪些文件/目录不要被跟踪（不纳入版本控制）”，也就是说.gitignore 文件会告诉 Git 哪些文件只放在我们的本地电脑，不提交到 Github 仓库中。

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_72_0.png)

如果你使用的是刘小排老师在深海圈提供的 raphael-starterkit-v1，就会看到项目中已经预置了一个.gitignore 文件，里边向 Git 声明了哪些文件不提交到 Github 仓库中。如果你看不懂这个文件配置了什么，可以让 Claude Code 给你解释一下。

你尤其要检查一下含有 API Key 等关键配置的 env 文件不要提交到仓库中。另外，再次提醒，前面第 3 部分“网站上线前的细节工作和注意事项”中，提到了要检查 API Key 是否硬编码，如果代码中存在 API Key，一定要做修改，统一把 API Key 放在配置文件中，并且不提交到 Github 仓库中。

### 2、在 Cursor 中提交项目到 Github

点击进入 Git 窗口：

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_74_0.png)

点击“Initialize Repository”初始化仓库：

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_74_1.png)

输入提交时的备注信息，然后点击“Commit”提交：

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_74_2.png)

接着点击“Publish Branch”推送代码到仓库：

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_75_0.png)

点击“private”这个选项，表示仓库私有，只有你自己可以看到，如果选择“public”，意味着公开仓库，别人也能看到你这个仓库的代码。这里要特别注意不要选错!

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_75_1.png)

提交成功后收到提交成功的提示。回到Github对应账号下查看这个项目:

### 3、在 GitHub Desktop 添加仓库

打开 GitHub Desktop，登录你的 Github 账号后，点击左上角这个向下的小三角形图标：

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_76_1.png)

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_77_0.png)

选择项目所在文件夹，然后点击“Add repository”添加：

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_77_1.png)

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_77_2.png)

这样，后续我们修改代码需要提交的时候，就可以很方便的通过 GitHub Desktop 而不需要使用 Git 命令进行代码提交了。

GitHub 和 GitHub Desktop 还有许多功能值得我们了解，上面是通过提交我们当前演示的这个 NextJS 项目为例做了一些简单的介绍，大家有兴趣可以一边实践一边通过 AI 进行学习。

## Vercel 一键部署 NextJS 网站

哥飞老师和小排老师都推荐我们使用 NextJS 框架，并不是因为其它编程语言不好，而是 Vercel+NextJS+Supabase 确实极大地降低了我们很多新手和小白上线一个网站的门槛。可能你没有注意过，NextJS 实际上是 Vercel 平台这个公司的一个开源产品，这种紧密的联系，使得我们部署一个 NextJS 网站到 Vercel 中会无比丝滑顺畅。我们直接看演示。

注意：你必须先注册好一个 Vercel 账号，并且连接放置你的项目代码的 Github 账号。

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_79_0.png)

![](img/dfbb2dfeb71bee05acaaa77b1203b1a0_79_1.png)

### 步骤 3：一键部署

### Step 3：解析域名：考虑 Vercel 已提供 SSL

Cloudflare 的 Nameservers 生效后，可以看到绿色的 “Active” 状态，这意味着我们可以开始在 Cloudflare 中做域名解析了。

我们回到 Vercel 的网站控制台，点击 “Settings” 中的 “Domains” 进入域名设置页面。

添加我们的域名，一般添加根域名和带 “www” 的二级域名，以我们这个网站为例，需要添加 “airobotlion.com” 和 “www.airobotlion.com”这两个域名。由于我习惯使用不带“www”的域名，因此没有选择把根域名跳转到“www”的域名上。

接下来，分别点击这两个域名的“Learn more”链接，可以看到 Vercel 已经告诉我们怎样解析这两个域名：不带 www 的根域名通过 A 记录解析到指定的 IP 地址，带 www 的二级域名通过 CNAME 解析到指定的域名。

回到 Cloudflare，点击“DNS”的“Records”添加域名解析记录，分别添加一条 A 记录 和一条 CNAME 记录 并分别保存。

这里特别注意，添加解析记录的时候，Cloudflare 默认会开启 “Proxy status” ，也就是你看到的橙色云朵。由于 Vercel 建议采用 DNS only（也就是灰色云朵），把 SSL 交由 Vercel 处理，需要 Cloudflare 加速/防护的子域再开代理并使用 Cloudflare 提供的 WAF/Rate Limiting/Bot Fight/缓存等能力。因此，我们这里添加解析记录的时候选择不开启 “Proxy status” ，避免叠加反向代理带来额外延迟并造成缓存管理冲突等问题。

添加完成后，可以看到一条 A 记录和一条 CNAME 记录，都显示灰色云朵。

回到 Vercel 网站控制台的域名管理页面，点击 “Refresh” 刷新，可以看到域名已经生效，并且显示 Vercel 自动给域名添加了 SSL。到了这一步，域名管理页面显示的这 3 个域名都可以打开我们的网站了。但是，一般情况下，我们只保留其中一个对外的域名，其它的需要重定向到对外的域名上，这样可以避免搜索引擎发现多个不同域名同时指向我们的网站首页。

比如，我想把“www.airobotlion.com”和“ai-robot-lion.vercel.app”这两个域名重定向到“airobotlion.com”这个域名上，对外只使用“airobotlion.com”这个域名，那么可以通过编辑域名，然后选择 301 重定向，把域名重定向到“airobotlion.com”这个域名上。反之，如果你想使用“www.airobotlion.com”这个域名，则把“airobotlion.com”和“ai-robot-lion.vercel.app”这两个域名重定向到“www.airobotlion.com”这个域名上。

现在我们已经完成了域名解析，可以通过“airobotlion.com”这个域名打开网站了，而且打开的时候，你会发现域名前面带有“https”，这就是 Vercel 给我们域名添加的 SSL。

你还可以尝试通过 “www.airobotlion.com” 和 “ai-robot-lion.vercel.app” 这两个域名打开我们的网站，会发现他们都会自动跳转到 “airobotlion.com” 这个域名上。

## 查看和跟踪你的网站流量情况和用户行为

网站上线之后，网站究竟有没有流量？有哪些用户访问了网站？他们在网站中做了哪些事情？有没有什么工具可以让我们洞悉网站的流量和用户的行为，为我们的出海助一臂之力？

这里简单概述这4个工具的作用：

- Google Search Console：跟踪并诊断你的网站在Google自然搜索中的表现与索引健康情况；
- Google Analytics：跟踪网站的各个渠道来源并分析用户行为与转化，回答“谁从哪里来、进站后做了什么”；
- Bing Webmasters：跟踪并诊断你的网站在Bing自然搜索中的表现与索引健康情况；
- Clarity：通过录像回放与热力图可视化用户行为，快速定位网站使用问题与转化问题。

## 通过 API 调试让 AI 更懂你的需求

前面第 4 部分中提到：我们在让 AI 帮我们编程时，要尽可能“想办法把话说清楚”。这一点刘小排老师做到了极致。他在开发一个 AI 产品时，会先查看 API 文档，然后通过 Postman 调试，然后把 API 文档、API 示例、示例输入和示例输出一并发给 AI，这样 AI 往往能更加明确他的需求，在他圈定的范围内给出更加合理的代码。

接下来，我们就调试一下 Nano-Banana 生图的 API，为后面 AI 更懂我们的需求并在网站中增加 Nano-Banana 生图功能做准备。

假设我们使用的是 Fal.ai 的 “fal-ai/gemini-25-flash-image/edit” 模型：

后面，我们在调试 API 的过程中需要使用到这个 Key。

### 步骤 2：构建这个 API 的 CURL 语句

你可以先查看 API 的文档，很多 API 文档中往往有各种语言版本的示例代码，找到其中 CURL 的那个。如果你找不到或者不确定，可以把 API 文档发给 ChatGPT，让它帮你构建适合在 Postman 工具中调试的 CURL 语句。

**ChatGPT 对话记录：**

> 请你联网阅读这份 API 文档：
> https://fal.ai/models/fal-ai/gemini-25-flash-image/edit/api
> 然后输出这个 API 通过 Postman 做 API 调试的 CURL

*ChatGPT 回复：*

下面是根据你给的文档整理出的、可直接被 Postman 导入的 curl 示例（含同步直返与队列三段式两种方式）。我已核对了模型输入字段与端点、认证与示例负载要求。

**同步直返（适合快速调试）**

> 直接拿到结果；注意同步端点是 https://fal.run/...

```bash
curl -X POST "https://fal.run/fal-ai/gemini-25-flash-image/edit" \
  -H "Authorization: Key $FAL_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "prompt": "make a photo of the man driving the car down the california coastline",
    "image_urls": [
      "https://storage.googleapis.com/falserverless/example_inputs/nano-banana-edit-input.png",
      "https://storage.googleapis.com/falserverless/example_inputs/nano-banana-edit-input-2.png"
    ],
    "num_images": 1,
    "output_format": "jpeg"
  }'
```

- 认证方式：在 `Authorization` 头中传 `Key $FAL_KEY`。
- 输入必填：`prompt`、`image_urls`；可选：`num_images`、`output_format`、`sync_mode`。

**队列三段式（适合长任务与回查）**

> 通过队列提交、查询状态、获取结果；端点基于 https://queue.fal.run/...

**1) 提交请求**

```bash
curl -X POST "https://queue.fal.run/fal-ai/gemini-25-flash-image/edit" \
  -H "Authorization: Key $FAL_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "prompt": "make a photo of the man driving the car down the california coastline",
    "image_urls": [
      "https://storage.googleapis.com/falserverless/example_inputs/nano-banana-edit-input.png",
      "https://storage.googleapis.com/falserverless/example_inputs/nano-banana-edit-input-2.png"
    ],
    "num_images": 1,
    "output_format": "jpeg"
  }'
```

> 成功会返回 `request_id` 与 `response_url`（形如 `/requests/<REQUEST_ID>`）。

**2) 轮询状态（可选）**

```bash
curl --request GET \
  --url "https://queue.fal.run/fal-ai/gemini-25-flash-image/edit/requests/<REQUEST_ID>" \
  -H "Authorization: Key $FAL_KEY"
```

> 当请求完成，相同 URL 的响应体会包含最终结果（与模型输出架构一致）。该路径样例与模式来源于官方 Queue 文档与其他模型示例。

**3) 获取结果（若上一步已返回最终结果可直接跳过）**

```bash
curl --request GET \
  --url "https://queue.fal.run/fal-ai/gemini-25-flash-image/edit/requests/<REQUEST_ID>" \
  -H "Authorization: Key $FAL_KEY"
```

**备注与要点**
- 模型输入结构（编辑任务必备）：`prompt`（编辑指令）、`image_urls`（要编辑的图片 URL 列表）；支持 `output_format: "jpeg" | "png"`；`sync_mode: true` 时可直接回传 data URI。
- 密钥放置：用 `Authorization: Key $FAL_KEY`，而不是 `Bearer`。
- Postman 使用：在 Postman 里选择 `Import → Raw text` 粘贴以上任一 curl，将环境变量 `FAL_KEY` 设为你的密钥即可发送。
- 官方示例与说明来源：本模型页面的 API/Schema、同步请求与队列端点文档。

ChatGPT 输出了“同步直返”和“队列三段式”两种 CURL 的调试方式。接下来，我们开始调试“队列三段式”的 CURL，因为 ChatGPT 在后面的沟通中建议在我们的 Nano-Banana 生成图片的网站中使用“队列三段式”的方式。

## Your traffic is almost ready to proxy

How proxying works

Your DNS records tell Cloudflare where your web content is hosted, so we can hide the origin and proxy your web traffic.

- Proxied: HTTP traffic goes through the Cloudflare network. Required for services like caching, smart routing, and Web Application Firewall.
- DNS only: HTTP traffic bypasses Cloudflare and goes directly to the origin, but DNS resolution still benefits from our fast and secure network. A gray cloud means proxying is off, no cloud means the record is not eligible for proxying.
